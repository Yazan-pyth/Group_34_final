**Цель:** Сформировать у обучающихся практические навыки проектирования детерминированных систем реального времени на базе FreeRTOS для обеспечения гарантированного времени отклика в задачах управления и ML-инференса на встроенных устройствах.

**Задачи:**

- Проанализировать влияние различных факторов (прерывания, приоритеты задач, блокировки) на детерминизм системы.

- Разработать архитектуру многозадачного приложения для отраслевого кейса (ТЭК или Транспорт) с гарантированным временем отклика.

- Подготовить отчёты по лабораторным работам Модуля 3.

## 2. Теоретические основы

### 2.1 Детерминизм в системах реального времени

Система реального времени характеризуется **детерминированным** поведением — способностью предсказать заранее, сколько времени займут задержки и отклики системы. Если система считается отказавшей при несоблюдении временных требований, она должна быть системой реального времени.

**Ключевые понятия:**

- **Детерминизм** — свойство системы, при котором время выполнения операций не зависит от внешних факторов (например, количества задач).

- **Real-time** — способность системы реагировать на события в заданные временные рамки.

### 2.2 FreeRTOS как ОСРВ

FreeRTOS — это ядро реального времени с открытым исходным кодом для встраиваемых устройств. Планировщик FreeRTOS использует **строгие приоритеты** для обеспечения детерминированного поведения в реальном времени: задачи могут быть назначены строгие приоритеты, и RTOS гарантирует, что задача с наивысшим приоритетом получает процессорное время.

**Основные механизмы FreeRTOS:**

- **Задачи (Tasks):** единицы выполнения с назначенным приоритетом.

- **Планировщик (Scheduler):** определяет, какая задача должна выполняться в данный момент.

- **Очереди (Queues):** для безопасной передачи данных между задачами и ISR.

- **Семафоры (Semaphores):** для синхронизации и управления доступом к ресурсам.

- **Таймеры (Timers):** для отсчёта времени и периодических действий.

### 2.3 Обеспечение детерминизма

Для обеспечения детерминированного времени отклика необходимо:

1.  **Правильное назначение приоритетов:**

    - Критические по времени задачи должны иметь наивысший приоритет.

    - Задача с наивысшим приоритетом выполняется до тех пор, пока она не завершится, не заблокируется или не будет вытеснена задачей с ещё более высоким приоритетом.

2.  **Минимизация времени в ISR (Interrupt Service Routines):**

    - ISR должны выполнять только минимально необходимые действия.

    - Обработка данных, полученных в ISR, должна выполняться в отдельной задаче с высоким приоритетом.

3.  **Управление блокировками:**

    - Использовать мьютексы с наследованием приоритета для предотвращения приоритетной инверсии.

4.  **Таймеры и временные задержки:**

    - FreeRTOS измеряет время с помощью счётчика тиков, который увеличивается при каждом прерывании таймера.

    - Задачи могут указывать максимальное время ожидания при блокировке.

## 3. Этапы выполнения

### Этап 1. Изучение теоретических основ FreeRTOS (2 ч)

**Задание:**  
Самостоятельно изучить архитектуру и принципы работы FreeRTOS.

**Ключевые вопросы для изучения:**

1.  **Управление задачами:**

    - Создание и удаление задач (xTaskCreate, vTaskDelete).

    - Состояния задач: Running, Ready, Blocked, Suspended.

    - Приоритеты задач и их влияние на планирование.

2.  **Планировщик:**

    - Вытесняющее планирование с фиксированными приоритетами.

    - Круговая диспетчеризация (round-robin) для задач одинакового приоритета.

3.  **Синхронизация и коммуникация:**

    - Очереди (xQueueSend, xQueueReceive).

    - Семафоры (бинарные, счётные).

    - Мьютексы и проблема приоритетной инверсии.

4.  **Управление временем:**

    - Системный тик и его влияние на детерминизм.

    - Таймеры программного обеспечения.

**Результат этапа:** Конспект (3–5 стр.) с описанием:

- Архитектуры FreeRTOS.

- Механизмов планирования.

- Методов синхронизации и коммуникации.

- Подходов к обеспечению детерминизма.

### Этап 2. Анализ факторов, влияющих на детерминизм (2 ч)

**Задание:**  
Изучить факторы, которые могут нарушить детерминированное поведение системы, и способы их минимизации.

**Ключевые факторы:**

| Фактор                            | Влияние на детерминизм                                                                | Способы минимизации                                                              |
|-----------------------------------|---------------------------------------------------------------------------------------|----------------------------------------------------------------------------------|
| **Прерывания (ISR)**              | Длительные ISR блокируют выполнение задач                                             | Минимизировать код в ISR, использовать bottom-half задачи                        |
| **Приоритетная инверсия**         | Низкоприоритетная задача блокирует высокоприоритетную                                 | Использовать мьютексы с наследованием приоритета                                 |
| **Динамическое выделение памяти** | Непредсказуемое время выполнения                                                      | Использовать статическое выделение памяти                                        |
| **Разное количество задач**       | Увеличение времени планирования (scheduler overhead) при большом числе активных задач | группировка функционала в одну задачу или использование битовых масок готовности |
| **Блокировки и ожидания**         | Задачи могут ждать неопределённое время                                               | Использовать таймауты при блокировках                                            |
| **Равные приоритеты**             | Круговая диспетчеризация вносит вариативность                                         | Минимизировать число задач с одинаковым приоритетом                              |

**Результат этапа:** Аналитическая записка (2–3 стр.) с:

- Описанием каждого фактора.

- Оценкой его влияния на детерминизм.

- Рекомендациями по минимизации.

### Этап 3. Решение кейса: обеспечение детерминированного времени отклика (3 ч)

**Задание:**  
Разработать архитектуру многозадачного приложения для выбранного отраслевого кейса с гарантированным временем отклика.

**Вариант А. ТЭК — система мониторинга насосного оборудования:**

- **Задачи:**

  - **Высокий приоритет (5):** Сбор данных с датчиков (вибрация, температура) по прерыванию от таймера (100 Гц).

  - **Средний приоритет (3):** Предобработка данных и детекция аномалий (TinyML-инференс).

  - **Низкий приоритет (1):** Передача данных по MQTT и логирование.

- **Требования к детерминизму:** Сбор данных должен выполняться с точностью ±1 мс, инференс — не более 50 мс.

**Вариант Б. Транспорт — система управления беспилотным погрузчиком:**

- **Задачи:**

  - **Высокий приоритет (5):** Обработка данных с LiDAR/камеры (50 Гц).

  - **Высокий приоритет (4):** Вычисление траектории и управление двигателями.

  - **Средний приоритет (2):** ML-детекция препятствий (RKNN-инференс).

  - **Низкий приоритет (1):** Телеметрия и логирование.

- **Требования к детерминизму:** Время отклика на препятствие \< 50 мс, управление двигателями — \< 20 мс.

**Требования к архитектуре:**

1.  **Назначение приоритетов:**

    - Критические задачи (сбор данных, управление) — наивысший приоритет.

    - ML-инференс — средний приоритет (может быть вытеснен критическими задачами).

2.  **Использование прерываний:**

    - Таймеры для генерации точных временных интервалов.

    - ISR минимальной длины — только фиксация события и отправка в очередь.

3.  **Синхронизация:**

    - Очереди для передачи данных от ISR к задачам.

    - Семафоры для синхронизации задач.

4.  **Управление временем:**

    - Использование таймаутов при ожидании в очередях.

    - Настройка частоты системного тика для требуемой точности.

**Результат этапа:** Архитектурная документация (3–5 стр.):

- Диаграмма задач с указанием приоритетов.

- Схема взаимодействия задач (очереди, семафоры).

- Описание обработки прерываний.

- Обоснование выбранных приоритетов и механизмов синхронизации.

- Оценка максимального времени отклика для каждого сценария.
