## Л4.1. Путь модели: PyTorch → ONNX → RKNN / TFLite Micro

### 1. При экспорте в ONNX встретилась unsupported op для целевого runtime. Корректная стратегия:

- **А)** Уменьшить размер входного изображения
- **Б)** Проигнорировать и продолжить деплой
- **В)** Увеличить число классов модели
- **Г)** Заменить оператор, переписать подграф или вынести его на CPU

### 2. Что обязательно проверить сразу после экспорта модели в ONNX?

- **А)** Только размер полученного файла
- **Б)** Только имя автора модели
- **В)** Только имя входного тензора
- **Г)** Совпадение выходов ONNX-модели с исходной на одинаковых входах

### 3. Почему для многих edge-runtime предпочтительны статические (фиксированные) формы входа?

- **А)** Они автоматически повышают точность
- **Б)** Они отключают квантование
- **В)** Они упрощают оптимизацию и статическое планирование памяти
- **Г)** Они увеличивают число поддерживаемых классов

### 4. Модель после конвертации в RKNN даёт немного другие результаты, чем FP32-оригинал. Наиболее вероятная причина:

- **А)** Имя файла модели изменилось
- **Б)** ONNX случайно переобучил модель
- **В)** Модель сменила число классов
- **Г)** Различия в реализации операторов, точности вычислений и предобработке

### 5. Динамические оси (dynamic axes) в ONNX могут быть проблемой на edge, потому что:

- **А)** Некоторые edge-runtime требуют фиксированных форм входа
- **Б)** Они всегда снижают точность
- **В)** Они не поддерживаются языком Python
- **Г)** Они увеличивают число классов

### 6. Что необходимо сохранять и переносить вместе с моделью для корректного деплоя?

- **А)** Только имя автора
- **Б)** Параметры предобработки, порядок классов и форму входа
- **В)** Только размер файла
- **Г)** Только дату обучения

### 7. Что такое operator fusion при оптимизации графа?

- **А)** Слияние нескольких датасетов
- **Б)** Объединение сигналов разных сенсоров
- **В)** Объединение нескольких операций в один оптимизированный узел
- **Г)** Слияние двух моделей в ансамбль

### 8. Часть графа выполняется на CPU вместо NPU, когда:

- **А)** Модель слишком точная
- **Б)** Используется формат INT8
- **В)** NPU не поддерживает некоторые операторы
- **Г)** Вход имеет фиксированную форму

### 9. Что означает numerical tolerance при валидации конвертированной модели?

- **А)** Допустимое отличие численных выходов от эталона
- **Б)** Допустимое число ошибок компиляции
- **В)** Частота интерфейса UART
- **Г)** Допустимая температура SoC

### 10. Типичный путь деплоя модели на Rockchip NPU:

- **А)** PyTorch → CSV → RKNN
- **Б)** RKNN → PyTorch → ONNX
- **В)** PyTorch → ONNX → RKNN
- **Г)** ONNX → CUDA → RKNN

## Л4.2. RKNN-Toolkit2, TFLite Micro, INT8-квантование и профилирование

### 1. Латентность одного инференса — 8 мс. Какой throughput при последовательной обработке (без конвейеризации)?

- **А)** Определить нельзя без размера модели
- **Б)** ≈8 инференсов в секунду
- **В)** ≈1000 инференсов в секунду
- **Г)** ≈125 инференсов в секунду

### 2. После PTQ в INT8 точность упала на 20%. Какой порядок диагностики корректен?

- **А)** Сразу отказаться от квантования и вернуться к FP32
- **Б)** Сменить имя выходного класса
- **В)** Увеличить размер корпуса устройства
- **Г)** Проверить предобработку, репрезентативность калибровки и чувствительные слои

### 3. Чем per-channel квантование обычно лучше per-tensor для свёрточных слоёв?

- **А)** Оно не требует никаких вычислений
- **Б)** Оно уменьшает число слоёв сети
- **В)** Оно учитывает различающиеся диапазоны значений по каналам и точнее
- **Г)** Оно всегда даёт меньший footprint

### 4. Чем QAT отличается от PTQ?

- **А)** QAT учитывает эффекты квантования уже во время обучения
- **Б)** QAT квантует только веса после экспорта
- **В)** QAT не требует калибровочного набора никогда
- **Г)** QAT применяется только к MCU

### 5. Почему latency обязательно измерять на целевом железе, а не оценивать теоретически?

- **А)** Потому что ПК не умеет считать время
- **Б)** Реальные драйверы, runtime и утилизация NPU не отражаются в теоретических оценках
- **В)** Чтобы изменить обучающий датасет
- **Г)** Теория всегда занижает результат ровно вдвое

### 6. Зачем перед замером latency делают несколько «прогревочных» (warm-up) запусков?

- **А)** Чтобы физически прогреть плату
- **Б)** Чтобы дообучить модель
- **В)** Чтобы стабилизировать runtime (кэши, JIT, аллокации) перед измерением
- **Г)** Чтобы увеличить точность модели

### 7. Что входит в memory footprint модели на устройстве?

- **А)** Только исходный код прошивки
- **Б)** Только обучающий датасет
- **В)** Веса, активации и рабочие буферы
- **Г)** Только размер ONNX-файла

### 8. Параметр scale при линейном квантовании определяет:

- **А)** Соотношение между целочисленным и исходным вещественным диапазоном
- **Б)** Число классов на выходе
- **В)** Размер входного изображения
- **Г)** Тактовую частоту NPU

### 9. Переход на INT4 не дал выигрыша по скорости. Наиболее вероятная причина:

- **А)** INT4 занимает больше памяти, чем FP32
- **Б)** Целевой ускоритель плохо или вовсе не поддерживает INT4-путь
- **В)** INT4 нельзя сохранить в файл
- **Г)** INT4 всегда точнее INT8

### 10. Что означает hardware-aware optimization?

- **А)** Оптимизация с учётом возможностей и ограничений конкретного ускорителя
- **Б)** Выбор самой новой модели из доступных
- **В)** Оптимизация только Python-обвязки
- **Г)** Оптимизация размера корпуса устройства
